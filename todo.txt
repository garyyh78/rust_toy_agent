# Rust Toy Agent Repo Quality Plan

Goal: turn this into a top-quality public GitHub repo that is easy to understand, easy to trust, easy to contribute to, and hard to accidentally break.

What the scan found:
- The repo already has good raw material: a real README, LICENSE, CONTRIBUTING guide, CI, lint config, and a non-trivial Rust test suite.
- The biggest gaps are consistency and public polish: stale docs/CI details, missing community health files, unclear release story, generated artifact noise, duplicated runtime paths, tool schema drift, and several large stringly-typed modules.
- The roadmap below is ordered. Earlier steps reduce confusion for everything after them.

## Phase 1: Product and repository contract

1. Define the repo contract and supported usage.
   Why this comes first:
   - The repo currently mixes "toy agent", benchmark harness, binary application, and reusable library concerns.
   - `Cargo.toml` exposes both library and binary surfaces, but the public promise is not explicit.
   Work:
   - Decide whether this project is primarily:
     1. a CLI application,
     2. a reusable Rust library,
     3. both, with a stable public API.
   - Decide whether crates.io publishing is part of the plan or whether the repo should be GitHub-only.
   - Decide which features are officially supported versus experimental: REPL, test mode, SWE-bench mode, teammates, worktrees, background tasks, skill loading.
   - Write down supported OSes, supported Rust version policy, and whether networked/live tests are part of the normal developer workflow.
Deliverables:
    - A short product-positioning section in `README.md`.
    - A compatibility/support section in docs.
    - A publish strategy in `Cargo.toml` (`publish = false` if app-only, or full package metadata if publishable).
    Done when:
    - A new visitor can tell within 30 seconds what the repo is for, who should use it, and how it is distributed.
    -> DONE: CLI app, GitHub-only, all features supported, publish=false, new quickstart-first README

2. Clean up repository artifacts and data ownership.
   Why this is critical:
   - Public repo quality drops fast when generated files, logs, binaries, and benchmark leftovers are mixed into source.
   - The scan found tracked gitlink entries under `scripts/swe_bench_data/*/repo` without a visible `.gitmodules`, plus generated outputs and duplicate test result files.
   Work:
   - Resolve the `scripts/swe_bench_data/*/repo` strategy:
     - If these are real submodules, add a correct `.gitmodules`, document setup, and make clone/bootstrap instructions reliable.
     - If these are not meant to be submodules, remove gitlinks and replace them with a fetch/bootstrap script plus cache location.
   - Untrack generated outputs and machine artifacts such as:
     - benchmark results in `scripts/swe_bench_results/`,
     - compiled test binaries in `task_tests/`,
     - duplicate result reports in `tests/TEST_RESULTS.md` and `tests/results/test_results.md`,
     - any workspace-generated directories that should stay local.
   - Tighten `.gitignore` so it matches actual policy.
   - Fix contradictory hygiene like tracking `.editorconfig` while also ignoring it.
   - Decide whether local/internal directories such as `.claude/`, `.tasks/`, `.worktrees/`, and `learning/` should be public-facing, documented, ignored, or moved.
Deliverables:
    - Clean root tree with source/docs/config only.
    - A documented policy for fixtures versus generated outputs.
    Done when:
    - `git status` stays clean after normal development and the GitHub file browser shows mostly source, docs, tests, and scripts instead of machine output.
    -> DONE: removed gitlinks, added fetch script, cleaned .gitignore, removed duplicate test results

## Phase 2: Public-facing documentation and onboarding

3. Rewrite the README around first-time user experience.
   Why this matters:
   - The current README is rich, but it leads with internal architecture instead of quickstart and demo value.
   - A top-quality GitHub repo wins on the first screen.
   Work:
   - Move a "60-second quickstart" to the top.
   - Add a short demo transcript showing a real session.
   - Add a short feature summary focused on user value, not only internals.
   - Keep the architecture/module map, but move it after install and usage.
   - Remove hard-coded counts and claims that drift over time, or derive them from code/build scripts.
   - Explain the safety model honestly: shell execution is powerful and should run in a sandboxed or disposable workspace.
Deliverables:
    - README sections in this order: what it is, quickstart, demo, install, usage, safety, architecture, testing, contributing.
    Done when:
    - Someone landing on the repo can run the project before they need to understand its internals.
    -> DONE: quickstart-first README with demo

4. Build a real docs structure instead of overloading the README.
   Why this matters:
   - Top-quality repos separate landing-page docs from deeper operational and architecture docs.
   - There is no `docs/` or `examples/` directory today.
Work:
    - Add `docs/` with focused pages such as:
      - `docs/quickstart.md`
      - `docs/configuration.md`
      - `docs/architecture.md`
      - `docs/testing.md`
      - `docs/security-model.md`
      - `docs/release-process.md`
      - `docs/benchmark-data.md`
    - Add `examples/` or transcript fixtures showing common usage.
    - Expand `CONTRIBUTING.md` so it reflects real commands and current workflow.
    - Remove or fix stale instructions such as `cargo e2e-sum` if that command does not exist.
    Deliverables:
    - A small docs tree with links from the README.
    Done when:
    - README stays concise, and deeper questions can be answered by linked docs without code-diving.
    -> DONE: added docs/{configuration,architecture,testing,security-model}.md

5. Fix package metadata and distribution metadata.
   Why this matters:
   - `Cargo.toml` currently has basic metadata only. It does not fully communicate discoverability or distribution intent.
   Work:
   - If GitHub-only: add `publish = false` and state installation via source or release binaries.
   - If publishable: add `homepage`, `documentation`, `keywords`, `categories`, and any `package.metadata.docs.rs` needed.
   - Add installation guidance for the chosen distribution path.
   - Make versioning policy explicit.
Deliverables:
    - A `Cargo.toml` that matches the repo contract from Step 1.
    Done when:
    - A maintainer or visitor does not have to infer whether this project is intended for crates.io, GitHub Releases, or source-only use.
    -> DONE: publish=false added

6. Add community health files.
   Why this matters:
   - The repo is missing basic GitHub trust and governance files expected of a polished public project.
Work:
    - Add `SECURITY.md` with private disclosure guidance and scope.
    - Add `CODE_OF_CONDUCT.md`.
    - Add `SUPPORT.md` or a short help/triage policy.
    - Add `CHANGELOG.md`.
    - Update `LICENSE` to include the actual copyright owner name.
    Deliverables:
    - All major community health files visible in the GitHub sidebar/community tab.
    Done when:
    - GitHub no longer reports missing community standards for the repo.
    -> DONE: SECURITY, CODE_OF_CONDUCT, SUPPORT, CHANGELOG, LICENSE with name

7. Add issue intake, PR guidance, and review ownership.
   Why this matters:
   - `.github/` currently only contains CI. There is no issue template, PR template, or CODEOWNERS.
Work:
    - Add issue templates/forms for:

      - question/support,
      - security note redirect to `SECURITY.md`.
    - Add `PULL_REQUEST_TEMPLATE.md` with test/docs/release checklist.

    - Optionally add label conventions and milestone guidelines.
    Deliverables:
    - A consistent intake path for issues and PRs.

    - New issues and PRs arrive with enough context to be actionable.
    -> DONE: issue templates, PR template, CODEOWNERS

8. Configure GitHub repository settings that are not stored in git.
   Why this matters:
   - Repo quality depends on settings as much as files.
   Work:
   - Enable branch protection on `main`.
   - Require CI checks before merge.
   - Require at least one review if that fits the workflow.
   - Enable Dependabot alerts and security updates.
   - Enable secret scanning and push protection if available.
   - Configure repository topics, description, homepage URL, and social preview image.
   - Decide whether Discussions or a Projects board should be enabled.
   Deliverables:
   - GitHub settings aligned with the repo's quality bar.
   Done when:
   - Merges, releases, and issue intake are constrained by consistent repo policy rather than personal discipline.

9. Add automated dependency maintenance.
   Why this matters:
   - There is no `dependabot.yml`, so crates and GitHub Actions can silently age.
Work:
    - Add Dependabot for Cargo.
    - Add Dependabot for GitHub Actions.
    - Group low-risk updates weekly.
    - Document how dependency updates are reviewed and tested.
    Deliverables:
    - `.github/dependabot.yml` plus a small maintenance policy.
    Done when:
    - Dependency freshness is automated instead of ad hoc.
    -> DONE: dependabot.yml for cargo + actions

## Phase 4: CI, verification, and release engineering

10. Rebuild CI so green actually means healthy.
   Why this matters:
   - The current CI has value, but it also has stale checks and trust gaps.
   - Examples from the scan: stale reference to `task_tests/sum_1_to_n/task.md`, no MSRV matrix despite `rust-version = 1.75`, `cargo audit` without visible install step, and an unpinned `shellcheck` action from `master`.
Work:
    - Fix stale assertions in the workflow.
    - Install and pin the Rust toolchain in every job, including `fmt`.
    - Test at least `1.75` and `stable` in CI.
    - Add `permissions`, `timeout-minutes`, and `concurrency` to workflows.
    - Pin all actions to trusted versions or SHAs.
    - Decide whether `clippy_pedantic` stays advisory or becomes required.
    - Add explicit docs build checks and release smoke checks.
    - Consider `cargo nextest` for faster and clearer test execution.
    Deliverables:
    - A CI pipeline whose failures are meaningful and whose success is trustworthy.
    Done when:
    - The repo can claim a verified MSRV, verified formatting/lints/tests, and stable workflow behavior.
    -> DONE: MSRV matrix, timeouts, perms, pinning, audit install fix

11. Add stronger security and supply-chain checks.
   Why this matters:
   - This project handles API keys and executes shell commands, so supply-chain and secret hygiene matter more than average.
   Work:
    - Keep `gitleaks`, but document allowlisting and false-positive handling.
    - Make `cargo audit` reliable.
    - Consider adding `cargo deny` for license/advisory policy.
    - Add checks for outdated or unused dependencies only if they are kept low-noise.
    - Review script safety and shellcheck scope.
   Deliverables:
    - CI checks that cover both code quality and dependency/security posture.
   Done when:
    - The repo has a repeatable, documented security baseline.
   -> DONE: gitleaks (v2), cargo audit (with install step), shellcheck (pinned to 2.0.0)

12. Create a real release process.
   Why this matters:
   - There are no tags, no changelog workflow, and no GitHub Releases story today.
   Work:
    - Adopt semantic versioning.
    - Maintain `CHANGELOG.md`.
    - Add a tag-triggered release workflow.
    - Build release binaries for supported platforms.
    - Upload binaries plus checksums.
    - Optionally generate SBOM/provenance if you want a best-in-class release trail.
    - Decide whether to use `cargo-dist`, custom workflows, or another release tool.
   Deliverables:
    - Version tags and downloadable GitHub Release artifacts.
   Done when:
    - Users can install a stable version without cloning the repo.
   -> DONE: release.yml workflow for tags v*, builds linux/darwin x64+arm64, checksums, auto-release

13. Make coverage and quality metrics actionable.
   Why this matters:
   - The workflow generates a coverage summary, but the policy around it is not clear.
   Work:
   - Decide whether coverage is informational or gated.
   - Store or upload coverage artifacts for inspection.
   - Track quality metrics that matter for this repo: test pass rate, coverage trend, benchmark stability, MSRV status, release health.
   Deliverables:
   - A documented quality bar instead of isolated CI steps.
   Done when:
   - Maintainers know what must stay green before merge or release.

## Phase 5: Architecture consistency and runtime cleanup

14. Collapse the duplicate agent runtimes into one supported path.
   Why this matters:
   - The scan found two agent engines: `src/agent_loop.rs` and `src/bin_core/agent_loop.rs`.
   - The REPL, test mode, and benchmark flows are not clearly guaranteed to use the exact same execution path.
   Work:
    - Choose one core agent loop implementation.
    - Extract shared logic into a small set of reusable modules.
    - Make REPL, `--test`, and SWE-bench all call that shared engine.
    - Delete or deprecate legacy paths once parity is proven.
   Deliverables:
    - One runtime model, one mental model, one place to fix bugs.
   Done when:
    - A behavior fix in the agent loop automatically applies to interactive and test execution.
   -> DONE: Migrated e2e_test.rs to use bin_core::agent_loop with State. bin_core/agent_loop now returns AgentLoopResult with tokens/rounds. Legacy src/agent_loop.rs still exists (169 tests rely on it) - can be deprecated after those tests migrate.

15. Replace the multi-source tool system with one typed registry.
   Why this matters:
   - The current tool story is fragmented across schema builders, a legacy raw `TOOLS` constant, multiple dispatch paths, and both `todo` and `TodoWrite` conventions.
   Work:
    - Define one authoritative tool registry.
    - Generate tool schemas from typed definitions.
    - Route all dispatch through one shared execution layer.
    - Remove legacy naming drift and make tool names consistent across docs, tests, and runtime.
    - Derive tool counts from the registry instead of hard-coding them in UI/docs.
   Deliverables:
    - One source of truth for tool metadata and dispatch.
   Done when:
    - Adding or changing a tool requires editing one place and tests immediately validate the full contract.
   -> DONE: Consolidated tool registry - full_agent_tools() derives from 25+ tool_* functions. dispatch_tool (async, State-based) is now the single dispatcher. Legacy TOOLS constant marked deprecated. Still pending: standardize "TodoWrite" -> "todo" naming.

16. Centralize command execution and enforce safety consistently.
   Why this matters:
   - `bash`, worktree operations, and background tasks currently rely on similar but not fully unified shell execution rules.
   - Safety today is partly a substring blocklist and not a robust execution policy.
   Work:
   - Create one command-execution module shared by foreground/background/worktree commands.
   - Enforce timeout, cwd, environment allowlist, and output capture consistently.
   - Make `config.rs` the single place for execution-related policy.
   - Be explicit about what is a best-effort guard versus what is a real safety boundary.
   - Improve logging so exit code, timeout, stdout, and stderr are clearly separated.
   Deliverables:
   - A single command runner with predictable behavior.
   Done when:
   - Shell-related behavior is uniform across all execution modes and failure modes are testable.

17. Improve error handling and remove runtime panic paths.
   Why this matters:
   - The codebase currently leans heavily on `String` errors and still has runtime `unwrap`/`expect` paths in important code.
   Work:
   - Introduce typed error enums per subsystem.
   - Use `anyhow` only at the app boundary if desired.
   - Replace runtime `unwrap`/`expect` with explicit error paths.
   - Return errors upward and only convert to process exit codes in `main`.
   - Standardize error rendering for CLI, logs, and tests.
   Deliverables:
   - Structured, testable errors across the app.
   Done when:
   - Unexpected user input, file corruption, and environment problems fail predictably instead of crashing or silently degrading.

18. Replace stringly JSON-heavy core flows with typed models.
   Why this matters:
   - Core logic currently depends on `serde_json::Value` in many places, which weakens compiler help and makes drift easier.
   Work:
   - Introduce typed structs/enums for message blocks, tool requests/results, task/worktree records, status enums, and selected persisted state.
   - Keep raw JSON only at the true API boundary.
   - Add serde round-trip tests and invariants for persisted models.
   Deliverables:
   - More compiler-checked behavior and fewer string-key bugs.
   Done when:
   - The highest-value flows are expressed in types, not ad hoc JSON indexing.

19. Refactor the god-modules into smaller units with sharper APIs.
   Why this matters:
   - Several files are very large and mix responsibilities: `src/agent_loop.rs`, `src/tools.rs`, `src/agent_teams.rs`, `src/tool_runners.rs`, `src/team_protocols.rs`, `src/context_compact.rs`.
   Work:
   - Split `tools.rs` into schema/registry/dispatch/execution concerns.
   - Split the agent loop into history validation, request building, model call, tool dispatch, and reminder policy.
   - Split team and protocol logic into message types, storage, protocol state, and orchestration.
   - Keep small unit tests close to the smaller modules.
   Deliverables:
   - Smaller review units and easier targeted testing.
   Done when:
   - Most modules have one clear job and new contributors can navigate the code without reading 800-900 line files.

20. Reduce the accidental public API surface.
   Why this matters:
   - `src/lib.rs` currently exports almost everything, including internal binary-oriented modules.
   Work:
   - Decide which library APIs are intentionally public.
   - Move binary-only internals out of the public crate surface.
   - Replace broad `pub` with `pub(crate)` where appropriate.
   - Add rustdoc for truly public items only.
   Deliverables:
   - A smaller and more intentional API boundary.
   Done when:
   - Internal refactors do not feel like public breaking changes.

## Phase 6: State integrity, persistence, and invariants

21. Harden persisted state handling.
   Why this matters:
   - Some persistence code currently defaults corrupted state to empty/default behavior instead of surfacing the problem.
   Work:
   - Treat missing files differently from malformed files.
   - Fail loudly or quarantine corrupted task/team/worktree data.
   - Add validation when loading persisted files.
   - Add migration/versioning if on-disk formats are expected to evolve.
   Deliverables:
   - Explicit persistence integrity rules.
   Done when:
   - Corruption becomes diagnosable instead of looking like "no data".

22. Tighten worktree correctness and git-state invariants.
   Why this matters:
   - The scan found weak invariants in worktree create/remove/update flows, including no-op success paths and incomplete `base_ref` handling.
   Work:
   - Make no-op updates explicit errors when they indicate drift.
   - Honor requested `base_ref` in worktree creation.
   - Ensure index state only changes after git operations succeed.
   - Verify remove/keep flows against real temporary repos.
   Deliverables:
   - A worktree subsystem that matches actual git state.
   Done when:
   - Worktree metadata and git reality cannot quietly diverge.

## Phase 7: Testing strategy and reproducibility

23. Expand integration and end-to-end testing around real risk areas.
   Why this matters:
   - The repo has many unit tests, but the highest-risk cross-module behaviors are not covered enough.
   Work:
   - Add integration tests for:
     - one shared runtime path used by REPL/test modes,
     - tool registry parity,
     - temporary git repo worktree lifecycle,
     - corrupted persisted state,
     - command timeout/cancellation,
     - metrics/log output shape.
   - Add regression tests for stale-doc/tool-count mismatches when practical.
   Deliverables:
   - Integration tests that protect the real system boundaries.
   Done when:
   - Major refactors can be made with confidence that the app still behaves the same way end-to-end.

24. Make tests hermetic by default and live only by opt-in.
   Why this matters:
   - Some tests and benchmark paths rely on external services, external repos, or extra tooling.
   Work:
   - Default to mocks/fixtures for normal CI and local `cargo test`.
   - Gate live API or network-heavy smoke tests behind explicit env vars or ignored tests.
   - Document prerequisites for benchmark and SWE-bench workflows.
   Deliverables:
   - Predictable, reproducible tests for contributors and CI.
   Done when:
   - A normal test run does not unexpectedly depend on internet access or special local state.

25. Strengthen pre-commit and local developer checks.
   Why this matters:
   - The current pre-commit hook blocks `.env` and format issues, which is a good start, but the local quality bar is still limited.
   Work:
   - Decide which checks belong in pre-commit versus CI.
   - Keep hooks fast and low-noise.
   - Document exactly how to install and bypass hooks safely when necessary.
   - Consider adding lightweight local checks for clippy/test subsets if they remain fast enough.
   Deliverables:
   - A local workflow that catches common issues early without punishing contributors.
   Done when:
   - Contributors can run one documented local command sequence and reliably match CI expectations.

## Phase 8: Observability, performance, and final polish

26. Improve logging, metrics, and operational debugging.
   Why this matters:
   - The repo already logs a lot, but a top-quality project also documents what logs mean and how to debug failures.
   Work:
   - Standardize structured log fields.
   - Document log locations, rotation/retention expectations, and privacy concerns.
   - Make metrics output format stable and documented.
   - Add troubleshooting docs for common failures: missing API key, network errors, invalid tool results, corrupted state, worktree failures.
   Deliverables:
   - A documented operational story for maintainers and users.
   Done when:
   - Bug reports can include enough information to diagnose issues quickly.

27. Add performance and stress verification where it matters.
   Why this matters:
   - Quality is not only correctness and docs; it is also reliable behavior under load or long sessions.
   Work:
   - Benchmark hot paths like message truncation, tool dispatch, context compaction, and persistence-heavy operations.
   - Add stress tests for long conversations, large tool outputs, and many task/worktree entries.
   - Track performance regressions only if the benchmarks are stable enough to trust.
   Deliverables:
   - A small, meaningful performance baseline.
   Done when:
   - Major refactors do not accidentally make common workflows much slower or less stable.

28. Do the final public-repo polish pass.
   Why this matters:
   - After the structural work is done, small finishing details matter a lot on GitHub.
   Work:
   - Add badges for CI, MSRV, license, and releases once URLs are stable.
   - Add a social preview image or terminal screenshot/GIF.
   - Clean the root directory so only high-value files remain visible.
   - Ensure naming is consistent everywhere: tool names, command names, docs, log labels, release artifacts.
   - Review every top-level file from the perspective of a first-time visitor.
   Deliverables:
   - A repo homepage that looks deliberate and well-maintained.
   Done when:
   - The first GitHub page view communicates quality without needing explanation.

## Suggested execution order

Start with these P0 items:
- Step 1: repo contract
- Step 2: artifact/data cleanup
- Step 3: README rewrite
- Step 6: community health files
- Step 10: trustworthy CI
- Step 14: unify runtime
- Step 15: unify tool registry

Then do these P1 items:
- Steps 4, 5, 7, 8, 9, 11, 12, 13, 17, 18, 19, 20, 21, 22, 23, 24

Finish with these P2 polish items:
- Steps 25, 26, 27, 28

Definition of success for the full roadmap:
- A new GitHub visitor can install and run the project quickly.
- The repo clearly communicates what is supported, how to contribute, and how to report issues securely.
- CI is trustworthy and verifies the documented support matrix.
- Release artifacts and changelogs exist.
- The runtime uses one core execution path.
- Tool definitions, docs, and UI no longer drift.
- High-risk behavior is protected by integration tests.
- The public repo view looks clean, intentional, and actively maintained.
